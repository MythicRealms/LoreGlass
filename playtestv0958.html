I Character Creation Engine

Covers the 3-step character builder, race presets, background bonuses, and initial save.

Create a new character — all 3 steps complete
  1. Launch the game. From the main menu, choose Create Character.
  2. Step 1 — Enter a name. Press Enter or click Next.
  3. Step 2 — Select a race. Confirm the preview card updates with correct stats and HP.
  4. Step 3 — Select a background. Confirm bonus stats and gold modifier display correctly.
  5. Click Create. Confirm you return to the main menu with the character name shown.
ExpectedCharacter name shown in main menu subtitle. .dat save file created in user:// directory.
Race stat presets apply correctly
  1. Create a Dwarf character. After creation, open the Stats HUD tab.
  2. Verify: Might 7, Finesse 3, Grit 8, Cunning 4, Charisma 3, Lore 3.
  3. Verify HP = 18, starting gold = 3000s (30g).
  4. Repeat with an Elf — Finesse 8, Lore 7, HP 10, gold 4000s.
ExpectedStats exactly match RACE_PRESETS in GlobalState. No rounding errors.
Background stat bonuses stack on top of race
  1. Create a Human Mercenary. Human base stats are all 5.
  2. Mercenary gives: Might +1, Grit +2, gold −500s.
  3. Open Stats tab. Verify Might = 6, Grit = 7, gold = 3000s (3500 − 500).
ExpectedStat bonuses applied on top of race. Gold modifier correctly subtracted.
Character limit enforced at 4
  1. Create 4 characters. Return to main menu.
  2. Attempt to create a 5th. The button should show MAX 4 REACHED and be disabled.
  3. Delete one character via the Select Character screen. Confirm the button re-enables.
Expected5th creation blocked. Deletion correctly frees a slot. File removed from disk.
Bug B2 fixed — delete now uses relative path. Verify the file is actually gone from user:// after deletion.
II Story Engine Content

Card loading, typewriter, choice conditions, stat rolls, and card-level effects.

Start story — loads start_card correctly
  1. From the library, open WakingRoad_v2.
  2. Click Begin / Start Over. Confirm the prose label fills with the start card text.
  3. Confirm the typewriter animation plays and the Skip button is visible.
  4. Click Skip. Confirm all text appears instantly and Skip button hides.
ExpectedTypewriter runs at configured speed. Skip reveals all text. Choice buttons appear after text loads.
Condition-gated choices — keyword check
  1. Find a card with a choice gated by a keyword condition the character does NOT have.
  2. Verify the button is either hidden (if hide_if_fails: true) or shown as disabled with the fail label in brackets.
  3. Use the dev card to grant the keyword. Return to that card and confirm the button now appears/enables.
ExpectedHidden buttons stay hidden. Disabled buttons show fail label. Met conditions enable the button.
Stat roll — success and failure paths
  1. Find a choice with a roll block (stat check).
  2. Click it. Verify the roll summary shows: die result, stat value, total vs DC, margin.
  3. Wait 3.5s. Confirm the correct card loads (success_goto or fail_goto).
  4. Test both outcomes if possible (reload / dev card to adjust stats).
ExpectedRoll display clearly shows all components. Correct card loads after delay. No double-navigation.
Save and resume — progress preserved
  1. Advance a few cards into the story. Note the current card ID (shown in pause menu).
  2. Press Menu → Main Menu.
  3. Return to the Library and choose the same book. Confirm it resumes at the correct card.
  4. Confirm gold, HP, and any keywords set during play are preserved.
ExpectedProgress resumes exactly where left off. All state intact.
Dev card — one-time claim guard
  1. Navigate to dev_card. Click Claim 500g + 3 potions.
  2. Verify gold and potions increase. Verify the button disappears (hidden by hide_if_fails).
  3. Navigate away and return to dev_card. Confirm the claim button is still gone.
  4. Confirm gold/potions did NOT increase again on re-entry (no card-level effect firing).
ExpectedClaim button hidden after use. Values do not increase on re-visits. DEV_GOLD_CLAIMED keyword set.
Content fix C1 — card-level effect removed. Verify no double-grant on re-entry.
III Inventory System New Feature

The core new system — instance-based inventory, snapshots, stackables, equip/unequip, direct sell.

give_item effect — item appears in inventory
  1. On any card, trigger an effect containing give_item. Use the dev card or a JSON card with:
// In a card's effect block or choice effect: { "give_item": { "id": "wolf_pelt", "qty": 3 } }
  1. Open the Items HUD tab. Confirm Wolf Pelt x3 appears under Trade Goods.
  2. Confirm the item shows its label, emoji, and sell price (20% of base).
ExpectedItem appears with correct qty. Stackable — one record, qty = 3. Sell price = floor(sell × 0.20).
Unique item instances — two swords are separate
  1. Give the character an Iron Sword twice via two separate give_item calls.
  2. Open Items tab. Confirm two separate Iron Sword entries appear, each with their own iid.
  3. Equip one. Confirm the equipped one shows ★ and the other stays unequipped.
  4. Confirm both persist after saving and reloading.
ExpectedTwo distinct instances. Only one can be equipped at a time. Both save correctly with separate iids.
Equip and unequip — combat stats update
  1. Note unarmed stats: DMG 2, +Hit 0.
  2. Give and equip a weapon (e.g. Iron Sword: DMG 6, +Hit 1).
  3. Check Main HUD — confirm DMG and +Hit update immediately.
  4. Open Equipment tab — confirm Main Hand slot shows the sword.
  5. Unequip it. Confirm stats revert to unarmed.
ExpectedStats reflect equipped item. Unequip reverts. Equipment tab shows correct slot states.
Direct sell from inventory — 20% value
  1. Give the character a wolf pelt (base sell: 80s).
  2. Open Items tab. Confirm sell link shows Sell 16s (80 × 0.20 = 16).
  3. Click sell. Confirm 16s added to gold. Confirm pelt removed from inventory.
  4. Attempt to sell an equipped weapon. Confirm it shows "Unequip to sell" message.
ExpectedSell price = floor(base_sell × 0.20). Item removed after sell. Equipped items blocked.
Old save migration — keywords become inventory
  1. Load a character save file that predates the inventory system (has HAS_IRON_SWORD in keywords, no inventory array).
  2. Confirm the game loads without error.
  3. Open the Items tab. Confirm the Iron Sword now appears as a proper inventory instance.
  4. Confirm trade good quantities from book_vars also migrated correctly.
ExpectedSeamless migration. Old saves work. No items lost. iid generated for each migrated item.
If no old saves are available, create one manually by editing a .dat file to remove the inventory key.

  3. Confirm HP bars for all enemies are shown with correct max HP.
  4. Confirm the player's HP bar and AC/DR are shown.
ExpectedInitiative line shown on round 0. All combatants visible with HP bars. Attack button present.
Attack roll uses equipped weapon stats
  1. Equip an Iron Sword (DMG 6, +Hit 1) before entering combat.
  2. Attack. Confirm the combat log shows the weapon name and correct DMG value.

ExpectedStatus shown in HUD. Damage/heal applies per tick. Expiry message shown. HUD clears.
Defeat → game over → save deleted

  2. Confirm the game over screen appears with the character's name.

IX Cross-Book Portability Cross-Book

Items, memories, and story keywords travelling between books. The core promise of the engine.

Item from Book A displays correctly in Book B
  1. In Book A, acquire an item defined in Book A's items.json only.
  2. Travel to Book B (which does not define this item). Open the Items HUD tab.
  3. Confirm the item still displays with its full label, emoji, stats, lore, and sell price.
  4. Confirm from: BookA appears on the item entry.
  5. Equip it if applicable. Confirm combat stats reflect it correctly.
ExpectedItem fully displayed from character snapshot. No "unknown item" fallbacks. Source book shown.
Same item id in two books — instances stay separate
  1. Define rusty_knife in Book A (DMG 2) and Book B (DMG 4) with different stats.
  2. Acquire the knife in Book A. Travel to Book B. Acquire the knife there too.
  3. Open Items tab. Confirm two separate rusty_knife entries.
  4. Confirm each shows its own stats (DMG 2 and DMG 4 respectively).
  5. Confirm the Book A knife still shows DMG 2 — Book B's definition did not overwrite it.
ExpectedTwo separate instances with independent stats. Snapshot wins over ITEM_LIBRARY. This is the core portability guarantee.
story_keywords and memories visible after cross-book travel
  1. In Book A, set a story_keyword and a memory_keyword.
  2. Travel to Book B. Open the Memories tab.
  3. Confirm both are still present, correctly formatted.
  4. Start a New Run of Book B. Confirm story and memory keywords still intact.
ExpectedCross-book memories and story keywords survive travel and New Run. Character carries their history.
